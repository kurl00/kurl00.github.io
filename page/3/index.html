DAMA（五）-数据建模和设计

Posted on 2022-05-17 Edited on 2022-06-13

数据建模和设计

数据建模是发现、分析和确定数据需求的过程，用一种称为数据模型的精确形式表示和传递这
些数据需求。

数据可以采用多种不同的模式来表示。其中最为常见的6种模式分别是：

  1. 关系模式，
  2. 多维模式、
  3. 面向对象模式、
  4. 事实模式、
  5. 时间序列模式
  6. NoSQL 模式。

业务驱动因素

目标和原则

数据建模的目标是确认和记录不同视角对数据需求的理解，从而使应用程序与当前和未来的业务需求更加紧密结合在一起，并成功的完成广泛的数据应用和管理活动奠定基础。

基本概念

  1. 数据建模和数据模型

    数据模型描述了组织已经理解或未来需要的数

  2. 建模的数据类型

    1. 类别信息
    2. 资源信息
    3. 业务事件信息
    4. 详细交易信息

  3. 数据模型组件

    who，what，when，where，why，how，measurement

    1. 实体别名
    2. 实体的图形表示
    3. 实体的定义
      1. 清晰
      2. 准确
      3. 完整
    4. 关系
      1. 关系的别名
      2. 关系的图形表示
      3. 关系的基数

  4. 数据建模的方法

    1. 关系建模
    2. 维度建模

  5. 数据模型级别

    1. 概念数据模型
    2. 逻辑数据模型
    3. 物理数据模型

  6. 规范化

    1. 第一范式。确保每个实体有一个有效主键，而且消除冗余分组，确保属性的原子性。
    2. 第二范式。确保每个实体有最小的主键，每个属性都依赖于完整主键
    3. 第三范式。确保每个实体都没有隐藏的主键，每个属性都不依赖于键值之外的任何属性
    4. Boyce/Codd范式。解决交叉的复合候选键问题。复合意味着不止一个，交叉是指键与键之间隐藏的业务规则
    5. 第四范式。所有三元关系分解为二院关系，直到这些关系不能再分解成更小的部分
    6. 第五范式。实体内部的依赖关系分解成二元关系，所有联结依赖部分主键

    通常要求达到第三范式水平即可，实践中后三项很少出现

  7. 抽象化

建立数据模型

1.正向工程

​ （1）概念数据模型建模

​ 1.选择模型类型，从关系、维度、基于事实或nosql的建模方法中选择一种

​ 2.选择表示方法。一旦选定建模的模式类型，就要考虑采用何种建模表示方法。如：信息工程法(IE)，对象角色建模(ORM)。

​ 3.完成初始概念模型

​ 4.收集组织中最高级的概念。包括时间、地点、用户、商品服务/交易

​ 5.收集与这些概念相关的活动。关系可以双向的，也可以涉及多个概念。

​ 6.合并企业属于。一旦获取某些用户观点，需要确保观点与企业术语和定义相一致。

​ 7.获取签署。初始模型完成后，确保对模型进行最佳实践及需求满足程度的评审。

1.多大程度上反应业务需求？ 要确保数据模型代表需求。

2.模型完整性？ 需求的完整性和元数据的完整性。需求完整性：意味着已经提出的每个需求都应在模型中得到满足。元数据的完整性，是指模型周围的所有描述性信息也要完整。

3.模型与模式的匹配程度？确保正在审查模型的具象级别和模式与该类型模型的定义相匹配

4.模型的结构如何？验证用于构建模型的设计实践，确保最终可以从数据模型构建数据库。

5.模型的通用性如何？评审模型的扩展性或者抽象程度。

6.模型遵循命名标准的情况如何？确保数据模型采用正确且一致的命名标准。

7.模型的可读性如何？确保数据模型易于阅读。

8.模型的定义如何？确保定义清晰、完整和准确

9.模型与企业数据架构的一致性如何？确认数据模型中的结构能否在更加广泛和一致的环境中应用，以便在组织中可以用一套统一的术语和模型结构。

10.与元数据的匹配程度如何？确认存储在模型结构中的数据和实际数据保持一致。

数据架构

从以下方面考虑数据架构

  1. 数据架构成果，包括不同层级的模型、定义、数据流
  2. 数据架构活动，用于形成、部署和实现数据架构的目标
  3. 数据架构行为，包括影响企业数据架构的不同角色间的协作、思维方式和技能

业务驱动因素

数据架构的主要职责：

  1. 利用新兴技术所带来的业务优势，从战略上帮助组织快速改变产品，服务和数据
  2. 将业务需求转换为数据和应用需求，确保能够为业务流程处理提供有效数据
  3. 管理复杂数据和信息，并传递至整个企业
  4. 确保业务和IT技术保持一致
  5. 为企业改革，转型和提高适应性提供支撑

数据架构成果和实施

包括：

  1. 数据存储和处理需求
  2. 设计满足企业当前和长期数据需求的结构和规划

架构师寻求一种能够为组织带来价值的方式对组织的数据架构进行设计。价值主要通过合适的技术应用、有效的运营、项目效率提升以及数据应用能力加强来体现。为了目标，要求组织有良好的设计和计划以确保设计和计划能被执行的能力。

数据架构师需要定义和维护：

  1. 定义组织中数据的当前状态
  2. 提供数据和组件的标准业务词汇
  3. 确保数据架构和企业战略及业务架构保持一致
  4. 描述组织数据战略需求
  5. 高阶数据整合概要设计
  6. 整合企业数据架构蓝图

数据架构实施包括：

  1. 使用数据架构构件（主蓝图）来定义数据需求，知道数据整合，管控数据资产，确保数据项目投入与企业战略保持一致
  2. 与参与改进业务或IT系统开发的利益相关方合作，学习并影响他们
  3. 通过数据架构及通用数据词汇，搭建企业数据语言

基本概念

  1. 企业架构类型

  1. 企业架构框架

  2. 企业数据架构

活动

简化数据和企业架构所面临的复杂问题，基于以下方式解决：

  1. 面向质量。专注于业务和IT开发周期内对数据架构进行不断改进
  2. 面向创新。专注于业务和IT转换，致力于新的期待和机会。创新性技术和数据使用驱动创新。

  3. 策略周期的。未来两年后期待使用的产品

风险有：

  2. 成功与否缺乏证据
  3. 缺乏管理者信任
  4. 管理层不正确的决策
  5. 文化冲击
  6. 缺乏有经验的项目经理
  7. 单一维度视角

组织和文化

组织接受并实施数据架构的能力依赖于：

  1. 对架构方法对接受度
  2. 确认数据属于组织的业务资产，而不仅仅是it任务
  3. 放弃局部数据视角，接受企业级数据视角能力
  4. 将架构交付成果整合到项目实施中的能力
  5. 规范数据治理的接受程度
  6. 立足企业全局，而不仅仅局限于项目交付成果和it解决问题的能力

数据架构治理

数据架构治理活动

  1. 项目监督
  2. 管理架构设计
  3. 定义标准
  4. 创建数据相关构件

度量指标

  1. 架构标准接受率
  2. 实施趋势
    1. 使用、重用、替代、废弃测量。决定使用新架构构件与重用，代替或废弃构件的比例
    2. 项目执行效率测量。测量项目的交付时间和可重用构件及知道构件的交付改进成本
  3. 业务价值度量指标
    1. 业务敏捷性改进。解释生命周期改进或改变的好处，改进延误成本的测量方法
    2. 业务质量。测量业务案例是否按期完成；基于新创建或集成的数据导致业务发生的改变，测量项目是否实际交付了这些变更
    3. 业务操作质量。测量改进效率的方法。实例包括准确性改进，时间减少，由于数据错误而导致的纠错费
    4. 业务环境改进。实例包括由于数据错误减少而改变的客户保留率和在地窖报告中当局评论的减少率

数据治理

##3.1 引言

业务驱动因素

  1. 减少风险
    1. 一般性风险管理。洞察风险数据对财务或商誉的影响，包括对法律和监管的响应
    2. 数据安全。通过控制活动保护数据资产，可获得性，可用性，完整性，连续性，可审计和数据安全
    3. 隐私。通过制度和合规性监控，控制私人信息，机密信息，个人身份信息等
  2. 改进流程
    1. 法规遵从性
    2. 数据质量提升。通过真实可信等数据提升业务绩效的能力
    3. 元数据管理。建立业务术语表，定义和定位组织中的数据；确保组织中数据繁多的元数据被管理和应用
    4. 项目开发效率。在系统生命周期中改进，以解决整个组织的数据管理问题，包括利用数据全周期治理来管理特定数据的技术债务
    5. 供应商管理。控制数据处理的合同，包括云存储、外部数据采购、数据产品销售和数据运维

数据治理与IT治理区分开 。 IT治理制定关于IT投资、IT应用组合和IT吸纳光明组合决策，从另一个角度还包括软硬件和总体技术架构。IT治理的作用是确保IT战略、投资与企业目标、战略的一致性。COBIT框架

##基本概念

以数据为中心的组织

以数据为中心的组织，将数据作为资产估值，在生命周期所有阶段进行管理，包括项目开发和持续运营阶段。业务处理的目标就是为了得到高质量数据。

数据治理组织

可参考政治治理角度来理解， 包括立法（定义策略，标准和企业架构），司法（问题管理和升级）和执法（保护和服务，管理职责）

通用的数据治理组织模型：

数据治理运营模型类型

集中管理模式中，数据治理组织监督所有业务领域中的活动

分布式管理模式中，每个业务单元中采用相同的数据治理运营模型和标准

联邦式管理模式中，数据治理组织与多个业务单元协同，以维护一致的定义和标准

数据管理职责

数据管理活动集中于以下：

  1. 创建和管理核心元数据
  2. 记录规则和标准
  3. 管理数据质量问题
  4. 执行数据治理运营活动

数据管理岗位的类型

  1. 首席数据管理专员。CDO替代角色，担任数据治理机构主席，也可以是虚拟组织或者在分布式数据治理组织中担任CDO，也可能是高层发起者

  2. 高级数据管理专员。数据治理委员会资深管理者

  3. 企业数据管理专员。负责监督跨业务领域数据职能

  4. 业务数据管理专员。业务领域专业人士，领域专家，对一个数据域负责。和利益相关方共同定义和控制数据

  5. 数据所有者。某个业务的数据管理专员，对其领域内数据有决策权

  6. 技术数据管理专员。某个知识领域内的IT专业人员，如：数据集成专家，数据库管理员，商务智能专家，数据质量分析师或元数据管理员

  7. 协调数据管理专员。领导并代表业务数据管理专员和技术数据管理专员进行跨团队或数据专员间的讨论

数据制度

包括数据治理管理的初衷的简要说明和基本规则，贯穿数据和信息的创造、获取、集成、安全、质量和使用的全过程。

数据制度是全局性的，支持数据标准以及数据管理和使用等关键方面的预期行为。

数据资产估值

是理解和计算数据对组织的经济价值的过程。

数据生命周期的大多数阶段涉及成本（获取、存储、管理和处置）。只有在使用时才有价值，同时还产生了与风险管理相关的成本。

当使用数据的经济效益超过了上述成本是时，就会显现价值。

其他度量价值的方式：

  1. 替换成本。灾难性数据破坏事件或数据中断时，数据替换或恢复成本，包括组织内的交易、域、目录、文档和指标信息等

  2. 市场价值。兼并或收购企业时作为企业资产的价值

  3. 发现商机。通过交易数据或通过售卖数据，从数据中发现商机获得的收入价值

  4. 售卖数据。一些组织为产品或销售将数据打包从数据中获得的洞察

  5. 风险成本。基于潜在罚款、补救成本和诉讼费用的估价。来自法律或监管的风险包括：a.缺少必要的数据；b.存在不应留存的数据; c.除上述成本外，包括数据不正确造成客户、公司财务和声誉受到损害; d.风险下降或风险成本的下降，其实是与提升和验证数据等操作干预成本等抵消之后的溢出部分。

数据管理知识领域内标准化概念示例：

  1. 数据架构。企业级数据模型，工具标准，系统命名规范
  2. 数据建模和设计。数据模型管理程序，数据模型命名规范，定义标准，标准域，标准缩写等
  3. 数据存储和操作。标准工具，数据库恢复和业务连续性标准，数据库性能，数据留存和外部数据采集
  4. 数据安全。数据访问安全标准，监控和审计程序，存储安全标准和培训需求。
  5. 数据集成。用于数据集成和数据互操作的标准方法、工具
  6. 文件和内容。包含内容管理标准及程序，包括企业分类法的使用，支持法律查询，文档和电邮保留期限，电子签名和报告分发方法
  7. 参考数据和主数据。数据管理控制流程，数据记录系统，建立标准及授权应用，实体解析标准
  8. 数据库仓库和商业智能。包括工具标准，处理标准和流程，报告和可视化格式标准，大数据处理标准
  9. 元数据。业务和技术元数据，元数据的集成和使用流程
  10. 数据质量。数据质量规则、标准测量方法、数据补救标准和流程
  11. 大数据和数据科学。数据源识别、授权、获取、记录系统、共享和刷新
