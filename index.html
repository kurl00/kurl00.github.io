数据集成和互操作

引言

数据集成是将数据整合成物理的或虚拟的一致格式。

数据互操作是多个系统间进行通信的能力。

数据集成和互操作的解决方案提供了大多数组织所以来的基本数据管理职能：

  1. 数据迁移和转换
  2. 数据整合到数据中心或数据集市
  3. 将供应商的软件包集成到组织的应用系统框架中
  4. 在不同应用程序或组织之间数据共享
  5. 跨数据存储库和数据中心分发数据
  6. 数据归档
  7. 数据接口管理
  8. 获取和接收外部数据
  9. 结构化和非结构化数据集成
  10. 提供运营智能话和管理决策支持

业务驱动因素

主要目的是为了对数据移动进行有效管理

目标和原则

目标：

  1. 及时以数据消费者所需的格式提供数据
  2. 将数据物理地或虚拟地合并到数据中心
  3. 通过开发共享模型和接口来降低管理解决方案的成本和复杂度
  4. 识别有意义的事件（机会和威胁），自动触发警报并采取相应行动
  5. 支持商务智能、数据分析、祝数据管理以及运营效率的提升

原则：

  1. 采用企业视角确保未来的可扩展性设计，通过迭代和增量交付实现
  2. 平衡本地数据需求与企业数据需求，包括支撑和维护
  3. 确保数据集成和互操作设计和活动的可靠性。

基本概念

1.抽取、转换、加载（ETL）

  1. 抽取。包括选择所需的数据并从其源数据中提取
  2. 转换。让选定的数据和目标数据库的结构相兼容
  3. 加载。在目标系统中物理存储或呈现转换结果
  4. 抽取、加载、转换（ELT）

  1. 映射。转换的同义词，从源结构到目标结构建立查找矩阵的过程，也是该过程的结果。

2.时延（Latency）

3.复制

4.归档

5.企业消息格式/规范格式

6.交互模型

7.数据集成和互操作架构概念

8.数据交换标准

数据建模和设计

数据建模是发现、分析和确定数据需求的过程，用一种称为数据模型的精确形式表示和传递这
些数据需求。

数据可以采用多种不同的模式来表示。其中最为常见的6种模式分别是：

  1. 关系模式，
  2. 多维模式、
  3. 面向对象模式、
  4. 事实模式、
  5. 时间序列模式
  6. NoSQL 模式。

业务驱动因素

目标和原则

数据建模的目标是确认和记录不同视角对数据需求的理解，从而使应用程序与当前和未来的业务需求更加紧密结合在一起，并成功的完成广泛的数据应用和管理活动奠定基础。

基本概念

  1. 数据建模和数据模型

    数据模型描述了组织已经理解或未来需要的数

  2. 建模的数据类型

    1. 类别信息
    2. 资源信息
    3. 业务事件信息
    4. 详细交易信息

  3. 数据模型组件

    who，what，when，where，why，how，measurement

    1. 实体别名
    2. 实体的图形表示
    3. 实体的定义
      1. 清晰
      2. 准确
      3. 完整
    4. 关系
      1. 关系的别名
      2. 关系的图形表示
      3. 关系的基数

  4. 数据建模的方法

    1. 关系建模
    2. 维度建模

  5. 数据模型级别

    1. 概念数据模型
    2. 逻辑数据模型
    3. 物理数据模型

  6. 规范化

    1. 第一范式。确保每个实体有一个有效主键，而且消除冗余分组，确保属性的原子性。
    2. 第二范式。确保每个实体有最小的主键，每个属性都依赖于完整主键
    3. 第三范式。确保每个实体都没有隐藏的主键，每个属性都不依赖于键值之外的任何属性
    4. Boyce/Codd范式。解决交叉的复合候选键问题。复合意味着不止一个，交叉是指键与键之间隐藏的业务规则
    5. 第四范式。所有三元关系分解为二院关系，直到这些关系不能再分解成更小的部分
    6. 第五范式。实体内部的依赖关系分解成二元关系，所有联结依赖部分主键

    通常要求达到第三范式水平即可，实践中后三项很少出现

  7. 抽象化

建立数据模型

1.正向工程

​ （1）概念数据模型建模

​ 1.选择模型类型，从关系、维度、基于事实或nosql的建模方法中选择一种

​ 2.选择表示方法。一旦选定建模的模式类型，就要考虑采用何种建模表示方法。如：信息工程法(IE)，对象角色建模(ORM)。

​ 3.完成初始概念模型

​ 4.收集组织中最高级的概念。包括时间、地点、用户、商品服务/交易

​ 5.收集与这些概念相关的活动。关系可以双向的，也可以涉及多个概念。

​ 6.合并企业属于。一旦获取某些用户观点，需要确保观点与企业术语和定义相一致。

​ 7.获取签署。初始模型完成后，确保对模型进行最佳实践及需求满足程度的评审。

1.多大程度上反应业务需求？ 要确保数据模型代表需求。

2.模型完整性？ 需求的完整性和元数据的完整性。需求完整性：意味着已经提出的每个需求都应在模型中得到满足。元数据的完整性，是指模型周围的所有描述性信息也要完整。

3.模型与模式的匹配程度？确保正在审查模型的具象级别和模式与该类型模型的定义相匹配

4.模型的结构如何？验证用于构建模型的设计实践，确保最终可以从数据模型构建数据库。

5.模型的通用性如何？评审模型的扩展性或者抽象程度。

6.模型遵循命名标准的情况如何？确保数据模型采用正确且一致的命名标准。

7.模型的可读性如何？确保数据模型易于阅读。

8.模型的定义如何？确保定义清晰、完整和准确

9.模型与企业数据架构的一致性如何？确认数据模型中的结构能否在更加广泛和一致的环境中应用，以便在组织中可以用一套统一的术语和模型结构。

10.与元数据的匹配程度如何？确认存储在模型结构中的数据和实际数据保持一致。

数据架构

从以下方面考虑数据架构

  1. 数据架构成果，包括不同层级的模型、定义、数据流
  2. 数据架构活动，用于形成、部署和实现数据架构的目标
  3. 数据架构行为，包括影响企业数据架构的不同角色间的协作、思维方式和技能

业务驱动因素

数据架构的主要职责：

  1. 利用新兴技术所带来的业务优势，从战略上帮助组织快速改变产品，服务和数据
  2. 将业务需求转换为数据和应用需求，确保能够为业务流程处理提供有效数据
  3. 管理复杂数据和信息，并传递至整个企业
  4. 确保业务和IT技术保持一致
  5. 为企业改革，转型和提高适应性提供支撑

数据架构成果和实施

包括：

  1. 数据存储和处理需求
  2. 设计满足企业当前和长期数据需求的结构和规划

架构师寻求一种能够为组织带来价值的方式对组织的数据架构进行设计。价值主要通过合适的技术应用、有效的运营、项目效率提升以及数据应用能力加强来体现。为了目标，要求组织有良好的设计和计划以确保设计和计划能被执行的能力。

数据架构师需要定义和维护：

  1. 定义组织中数据的当前状态
  2. 提供数据和组件的标准业务词汇
  3. 确保数据架构和企业战略及业务架构保持一致
  4. 描述组织数据战略需求
  5. 高阶数据整合概要设计
  6. 整合企业数据架构蓝图

数据架构实施包括：

  1. 使用数据架构构件（主蓝图）来定义数据需求，知道数据整合，管控数据资产，确保数据项目投入与企业战略保持一致
  2. 与参与改进业务或IT系统开发的利益相关方合作，学习并影响他们
  3. 通过数据架构及通用数据词汇，搭建企业数据语言

活动

简化数据和企业架构所面临的复杂问题，基于以下方式解决：

  1. 面向质量。专注于业务和IT开发周期内对数据架构进行不断改进
  2. 面向创新。专注于业务和IT转换，致力于新的期待和机会。创新性技术和数据使用驱动创新。

生命周期预测

  1. 当前的。 当前支持和使用的产品

  2. 部署周期的。 未来1～2年内部署使用的产品

  3. 策略周期的。未来两年后期待使用的产品

  4. 退役的。一年内，组织已经停止使用或打算停止使用的产品

  5. 优先的。被多数应用优先使用的产品

  6. 限制的。在一定应用中限制使用的产品

  7. 新兴的。 为将来可能的部署研究和试行的产品

  8. 审核的。已评估的产品，评估结果目前不能用于以上状态的产品

实施指南

就绪评估和风险评估

风险有：

  1. 缺少管理层支持
  2. 成功与否缺乏证据
  3. 缺乏管理者信任
  4. 管理层不正确的决策
  5. 文化冲击
  6. 缺乏有经验的项目经理
  7. 单一维度视角

组织和文化

组织接受并实施数据架构的能力依赖于：

  1. 对架构方法对接受度
  2. 确认数据属于组织的业务资产，而不仅仅是it任务
  3. 放弃局部数据视角，接受企业级数据视角能力
  4. 将架构交付成果整合到项目实施中的能力
  5. 规范数据治理的接受程度
  6. 立足企业全局，而不仅仅局限于项目交付成果和it解决问题的能力

数据架构治理

数据架构治理活动

  1. 项目监督
  2. 管理架构设计
  3. 定义标准
  4. 创建数据相关构件

度量指标

  1. 架构标准接受率
  2. 实施趋势
    1. 使用、重用、替代、废弃测量。决定使用新架构构件与重用，代替或废弃构件的比例
    2. 项目执行效率测量。测量项目的交付时间和可重用构件及知道构件的交付改进成本
  3. 业务价值度量指标
    1. 业务敏捷性改进。解释生命周期改进或改变的好处，改进延误成本的测量方法
    2. 业务质量。测量业务案例是否按期完成；基于新创建或集成的数据导致业务发生的改变，测量项目是否实际交付了这些变更
    3. 业务操作质量。测量改进效率的方法。实例包括准确性改进，时间减少，由于数据错误而导致的纠错费
    4. 业务环境改进。实例包括由于数据错误减少而改变的客户保留率和在地窖报告中当局评论的减少率
